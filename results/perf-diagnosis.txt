Performance Diagnosis: Runaway Event Loop in Karpenter Provisioning/Consolidation
==================================================================================

Root Cause Analysis
-------------------

Three issues combined to cause unbounded event generation:

1. PROVISIONING LOOP OVER-SCHEDULING (handler.rs)
   The ProvisioningHandler re-scheduled itself whenever `!decisions.is_empty()`,
   meaning every time it provisioned nodes it would fire again 5s later even if
   all pending pods were already addressed by the current decisions. This created
   unnecessary provisioning cycles that could interact with consolidation.

   Fix: Only re-schedule if pending_queue has more pods than were addressed by
   current provisioning decisions.

2. CONSOLIDATION LOOP UNCONDITIONAL RE-SCHEDULING (consolidation.rs)
   The ConsolidationHandler always re-scheduled the next consolidation loop
   regardless of whether any nodes existed. Combined with the provisioning
   feedback loop, this created a steady-state oscillation:
     consolidation evicts pods → pods go pending → provisioning launches nodes →
     pods scheduled on new nodes → consolidation evaluates again → repeat

   Fix: Only re-schedule consolidation if there are nodes to evaluate.

3. NO SAFETY LIMIT ON run_to_completion (engine lib.rs)
   Engine::run_to_completion had no upper bound on events processed. Any
   feedback loop between handlers would run indefinitely.

   Fix: Default cap of 1,000,000 events. Added run_to_completion_with_limit()
   for explicit control.

Dominant Event Type
-------------------
KarpenterProvisioningLoop and KarpenterConsolidationLoop events dominated,
with each cycle generating NodeLaunching/NodeCordoned/NodeTerminated follow-ups
that further inflated the queue.

Changes Made
------------
- crates/kubesim-engine/src/lib.rs:
    run_to_completion() now caps at 1M events (safety valve)
    Added run_to_completion_with_limit() for explicit limits

- crates/kubesim-karpenter/src/handler.rs:
    ProvisioningHandler only re-schedules if unaddressed pending pods remain

- crates/kubesim-karpenter/src/consolidation.rs:
    ConsolidationHandler only re-schedules if nodes exist to evaluate

Verification
------------
All 67 existing tests pass. The e2e_simulation test (which exercises the full
provisioning → consolidation → metrics pipeline) completes in <1s.
